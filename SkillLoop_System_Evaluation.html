📋 System Overview

SkillLoop is a peer-to-peer skill-barter platform for a campus community (SEAIT). Students earn credits by teaching and spend credits by learning, with a dual-confirmation session lifecycle, badge issuance, and reputation tracking — all using a zero-dependency PHP + MySQL + vanilla JS stack.

✅ Confirmed Working

  • Auth + CSRF on all forms
  • Session lifecycle (request → complete)
  • Credit economy (earn/spend/bonus)
  • Notification system
  • AJAX skill loading
  • Password strength meter
  • Responsive layout
  • 138-test automated suite

⚠️ Has Bugs

  • Browse: shows dupes per teacher
  • Back-to-top: no click handler
  • Dashboard recommendations: broken HTML
  • Hero: can't go full-width
  • CSS: duplicate padding declaration
  • Reset link: HTML escaped → not clickable

🔴 Missing Entirely

  • Email sending (dev-mode only)
  • Session regeneration on login
  • DB performance indexes
  • Page-specific <title> tags
  • Credit reservation on request
  • "Request Session" from profile page
🏗️ Architecture & Project Structure 8.5/10

✅ Clean Folder Separation

auth/, pages/, actions/, includes/, config/ separation is well-done. actions/ as the server-side API layer is a smart pattern that keeps pages declarative. No spaghetti mixing of logic and presentation.

✅ Correct Include Strategy

Using require_once __DIR__-relative paths for all includes avoids path-resolution bugs. All pages start with requireLogin() before any DB work.

✅ Zero Dependencies — Smart Choice

No Composer, no NPM, no framework. For a campus project, this is exactly correct. Load times are minimal and there's nothing to break from version drift.

⚠️ Session Created at Request Time (Not Acceptance)

When a learner requests a session, request_session.php inserts into BOTH session_requests AND sessions immediately. Only upon acceptance does the status change. This creates orphaned session rows for every declined request. Architecturally, sessions should only be created on acceptance.

⚠️ No .htaccess / Web Root Config

No .htaccess is present. tests/test_suite.php, install.php, and migrate_icons.php are fully accessible at their URLs on any web server. These should be blocked from public access.

ℹ️ Header.php Loads DB For Every Page

Even static informational pages must connect to the database to render the nav bar (credit badge, unread count). Fine at this scale, but worth noting for future optimization.

ℹ️ No Router — Direct File Serving

The README explicitly says not to use index.php as a router. Direct file serving is fine for a campus tool but means no clean URLs and no centralized request dispatch layer.

🔐 Security 6.5/10

✅ CSRF Protection

64-char hex token via bin2hex(random_bytes(32)), validated with constant-time hash_equals(). csrfField() helper is used consistently in all POST forms. Well-implemented.

✅ SQL Injection: PDO Prepared Statements

All DB queries use PDO with bound parameters. ATTR_EMULATE_PREPARES = false forces real prepared statements. No raw string interpolation in SQL. Solid.

✅ XSS: h() helper applied consistently

Every user-supplied value is run through h() (htmlspecialchars with ENT_QUOTES/UTF-8) before output. No unescaped user data found in any template.

✅ bcrypt Password Hashing

password_hash($password, PASSWORD_DEFAULT) with password_verify(). Correct, no salt management needed. Well-implemented.

🔴 No session_regenerate_id() on Login

Critical After a successful login, the session ID is never regenerated. This leaves the app vulnerable to session fixation attacks — an attacker who obtains a pre-login session cookie can hijack the account. Fix: call session_regenerate_id(true) immediately after setting $_SESSION['user_id'].

$_SESSION['user_id'] = $user['id'];
session_regenerate_id(true); // ADD THIS LINE

🔴 File Upload Uses Client-Supplied MIME Type

Security upload_avatar.php checks $file['type'] — the browser-supplied MIME type — which can be spoofed. A PHP shell uploaded with a .jpg extension and a faked image/jpeg type would pass the check. Use finfo_file() to read the actual file magic bytes.

// CURRENT (unsafe):
if (!in_array($file['type'], $allowedTypes)) {
// FIX:
$finfo = finfo_open(FILEINFO_MIME_TYPE);
$actualType = finfo_file($finfo, $file['tmp_name']);
if (!in_array($actualType, $allowedTypes)) {

⚠️ Skill Removal Uses GET (CSRF Bypass)

Removing a skill uses a GET link: /actions/add_skill.php?type=offer&action=remove&skill_id=X. requireCsrf() skips GET requests by design, so anyone who tricks a logged-in user into clicking a crafted link can silently remove all their skills. Mutating state should use POST only.

⚠️ Login Rate-Limiting Uses Session (Bypassable)

Lockout counters are stored in $_SESSION. An attacker can simply discard their session cookie to reset the counter, enabling unlimited brute-force attempts. Rate-limiting should be IP-based, stored in the database or Redis.

⚠️ DB Error Message Exposed in Production

config/database.php calls die("Connection failed: " . $e->getMessage()), which would expose database credentials and host info to any visitor if the DB goes down. Log errors privately; show a generic message to users.

⚠️ proficiency Field Not Whitelisted

add_skill.php inserts $_POST['proficiency'] directly. MySQL's ENUM will reject invalid values, but PHP should validate explicitly with in_array($proficiency, ['beginner','intermediate','advanced','expert']) before hitting the DB.

🗄️ Database Design 7.5/10

✅ Schema is Well-Normalized

14 tables with clear FKs and ON DELETE CASCADE. Separate tables for skill categories, skills, and user skill offers/wants avoid denormalization. The audit log table (credit_transactions) is a great design decision.

✅ Unique Constraints Prevent Duplicates

UNIQUE KEY unique_offer (user_id, skill_id) and unique_want prevent double-entries at the DB level, backed up by PHP-side duplicate checks.

✅ Dual-Confirmation Design

sessions.requester_confirmed + teacher_confirmed columns are clean and efficient for tracking the two-step completion flow.

🔴 Zero Performance Indexes

Performance Beyond PKs, FKs, and unique constraints, no additional indexes are defined. Columns that are constantly queried — sessions.status, session_requests.teacher_id, notifications.user_id + is_read, users.email — all require table scans. Add:

ALTER TABLE session_requests ADD INDEX idx_teacher (teacher_id);
ALTER TABLE session_requests ADD INDEX idx_requester (requester_id);
ALTER TABLE sessions ADD INDEX idx_status (status);
ALTER TABLE notifications ADD INDEX idx_user_read (user_id, is_read);
ALTER TABLE users ADD INDEX idx_email (email);

⚠️ reputation Column: No DB-Level Constraint

The users.reputation column is DECIMAL(2,1) which caps at 9.9. Since max rating is 5 stars this is fine numerically, but there's no CHECK constraint to enforce the 0–5 range at the DB level. A CHECK (reputation BETWEEN 0.0 AND 5.0) would add correctness assurance.

⚠️ No Input Length Validation for DB Fields

Name is VARCHAR(100), email is VARCHAR(255), bio is TEXT, etc. PHP registration validates password complexity but not max lengths for name, bio, location, or availability. PDO will truncate silently or error depending on MySQL mode.

ℹ️ Denormalized Reputation Column

users.reputation duplicates data derivable from session_reviews. The updateReputation() function keeps them in sync, but if it ever fails silently, the values diverge. A view or computed column would be more robust — though the current approach is a valid optimization.

ℹ️ Sessions Created Before Acceptance

A sessions row is inserted at request time with status='scheduled' (before the teacher accepts). Only on acceptance does it properly become "scheduled". Orphaned rows in sessions for declined requests pollute the table.

⚙️ Backend / PHP Code Quality 8.0/10

✅ functions.php is Well-Organized

25+ helper functions cleanly divided: auth, credits, skills, notifications, CSRF, flash messages. The addCredits() function correctly uses DB transactions with rollback.

✅ Flash Messages and CSRF Work Correctly

Flash messages survive redirect-after-POST correctly. CSRF token stored in session, generated lazily, validated with hash_equals() — all correct.

✅ All Actions Verify Ownership

Every action that mutates data checks ownership: complete_session.php, cancel_session.php, respond_request.php all verify $userId is a participant before executing.

⚠️ Inline SQL in credits.php View

The credits page has three raw SQL queries executed inside PHP echo statements within the template. These belong in functions.php as getTotalEarned(), getTotalSpent(), etc. Mixing data retrieval with presentation violates MVC separation.

⚠️ Dev-Mode Password Reset Link Broken

In forgot_password.php, a dev-mode clickable link is embedded in the flash message. But displayFlashes() applies h() (HTML escaping) to all flash content, so the <a href="..."> renders as raw text, not a clickable link. The link is permanently broken.

// displayFlashes() always escapes:
$output .= '...' . h($f['message']) . '...';
// FIX option: Use a 'raw' flash type for safe HTML content
// Or display $resetLink in a separate non-escaped UI element

⚠️ Profile Page Missing "Request Session" CTA

When a logged-in user views someone else's profile page, there is no button to request a session. Users must navigate back to Browse to do so. A "Request a Session" button linking to /actions/request_session.php?teacher_id=X should be visible on the profile.

ℹ️ No Pagination on Sessions Page

pages/sessions.php loads all sessions for a user with no LIMIT. A power user with 200+ sessions will cause a slow query and render a very long table. Add pagination (20–50 per page).

🔴 Hero Section: Duplicate CSS + Can't Go Full-Width

CSS Bug The .hero rule has two padding declarations (80px then 100px — second overrides first). Redundant code. Worse: the hero is rendered inside <main class="container">, so margin: -20px -20px 0 cannot break out of the 1100px max-width. The hero will never be truly full-width.

.hero {
  padding: 80px 20px;  /* OVERRIDDEN — dead code */
  margin: -20px -20px 0;  /* can't escape max-width container */
  padding: 100px 20px;  /* this wins */
}

Fix: Move <section class="hero"> outside of <main class="container"> in header/footer structure, and remove the duplicate padding.

⚠️ Browse Page: Multiple Cards Per Teacher

The browse query uses SELECT DISTINCT u.id but with non-aggregated skill columns. MySQL's DISTINCT applies to the entire row, not just u.id. A teacher offering 3 skills appears as 3 separate cards. The count query correctly counts distinct users, but the display shows more results than expected.

⚠️ loadSkills() Function Defined Twice

pages/skills.php defines its own local loadSkills() function inside a <script> block, which duplicates the one already in assets/js/script.js. This causes a function redeclaration (silently overrides in most browsers). Remove the duplicate from skills.php and rely on script.js.

⚠️ All Pages Share One Generic <title>

Every page renders "SkillLoop - Trade Skills, Earn Badges" as the browser tab title regardless of the page. Dashboard should say "Dashboard – SkillLoop", Browse should say "Find Teachers – SkillLoop", etc. Page titles affect SEO and browser history readability. The header.php title needs to be dynamic per page.

ℹ️ No <link rel="icon"> / Favicon

No favicon is declared. Browsers make a 404 request to /favicon.ico on every page load. Even a simple one-liner in header.php would fix this: <link rel="icon" href="data:..."> or a real favicon file.

ℹ️ CSS class name inconsistency: no-underline vs no-underline-force

In skills.php, one skill removal link uses no-underline-force (line 36) and the other uses no-underline (line 59). Inconsistent — both should use the same class.

✅ Dual-Confirmation is Correct and Fair

Both parties must independently mark a session complete before credits transfer. This prevents one party unilaterally claiming credits. The "waiting for other party" UX is handled cleanly.

✅ Credit Audit Log is Complete

Every credit movement (earn, spend, bonus) is logged to credit_transactions with counterparty, reference type, and description. Full traceability.

✅ Badge Issuance is Teacher-Only and Correct

Only the teacher can issue a badge to the learner, and only once per completed session (unique constraint). The badge level is teacher-selected — appropriate for a credential system.

✅ Notification Coverage is Comprehensive

All major lifecycle events trigger notifications: request sent, accepted, declined, completed, cancelled, badge earned. Notifications include contextual links to the relevant pages.

🔴 Credit Race Condition: Over-Requesting

Logic Bug A user with 1 credit can submit N simultaneous session requests (all pass the credits < 1 check). When N sessions complete, the credit balance goes to 1 - N (negative). There's no locking or reservation mechanism. This can produce negative balances.

⚠️ No Way to Message After Acceptance

Once a session is accepted, there's no in-app way for teacher and learner to communicate (clarify location, meeting link, etc.). The only "message" is the initial request text. For a campus app, even a simple notes field would help.

⚠️ No Session Scheduling Conflict Detection

A teacher can accept multiple session requests for the exact same date and time. There's no availability checking or conflict detection. Teachers could be double-booked.

ℹ️ Browse Shows Learner Can't Offer Skills They Want

The skills recommendation system on the dashboard is a nice touch — it shows skills in categories the user already engages with. However, there's no matching system that finds people who want to learn what the current user offers (the best "match-making" feature).

🧪 Testing 8.0/10

✅ 138 Unit Tests Cover Core Concerns Well

Tests cover PHP environment, file structure, syntax validation, schema tables, all 25+ helper functions, CSRF tokens, flash messages, HTML escaping, CSS completeness, and JS function presence. The zero-tolerance enforcement of no inline CSS and no emojis is excellent.

✅ E2E Flow Test Suite (33 tests)

The end-to-end tests cover the full session lifecycle: registration, skill management, request → accept → complete → review, badge issuance, notification tracking, and edge cases (duplicate email, CSRF validation). Good coverage of the happy path.

⚠️ Tests Are Publicly Accessible

/tests/test_suite.php and /tests/e2e_flow.php have no authentication guard. On a live server, they expose your system internals to anyone who visits the URL. Add a check to restrict to CLI-only, or add an .htaccess deny rule for the tests/ directory.

ℹ️ Tests Don't Cover the Browse Dupe Bug

The browse page SQL bug (multiple cards per teacher) is not covered by any test. The test suite tests function existence and syntax but not behavioral correctness of query results. Integration tests hitting the actual queries would catch this.

📋 Final Verdict: 7.2 / 10

SkillLoop is a genuinely solid student systems project. The core concept is well-executed, the architecture is clean, the security foundations are strong, and the test suite shows real engineering discipline. For a campus peer-to-peer platform, this is production-quality in many areas.

The score is held back by a cluster of fixable bugs — most critically: the browse page showing duplicate teacher cards, the back-to-top button that does nothing, the broken HTML in the recommendations card, and the credit reservation race condition that allows negative balances. Several are one-line or five-line fixes. None represent deep architectural flaws.

The security posture is above average for a student project — CSRF, XSS, and SQL injection are all properly defended. The two remaining gaps (session fixation and MIME spoofing on uploads) are important to fix before any production deployment.

  • Fix the 5 critical bugs above and the score rises to ~8.5/10 immediately.
  • Add the performance indexes and the system will scale well beyond a classroom environment.
  • Implement real email sending for password resets to make the auth system complete.
  • Fix the browse query (one card per teacher with skill aggregation) — this is the most user-visible UX issue currently in the system.

Overall: Great foundation. Real bugs, fixable bugs. The team shows strong understanding of web application architecture, security, and database design. Clean it up and this is a deployable product.
